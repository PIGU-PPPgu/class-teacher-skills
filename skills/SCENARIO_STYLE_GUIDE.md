# 场景文档与文件排版规范

这个规范用于统一 `class-assistant` 及后续班主任 skill 的场景组织方式。

目标不是把内容写得更长，而是让：

- 老师一眼知道怎么用
- agent 一眼知道先读什么
- 模板、示例、脚本长得一致
- 新增场景时不会越写越乱

---

## 1. 场景目录标准结构

每个场景建议放在独立目录，统一使用下面结构：

```text
scenarios/
  01-parent-daily-notice/
    SCENARIO.md
    templates.md
    examples/
      input.sample.md
      output.sample.md
    scripts/          # 可选
    assets/           # 可选
```

### 必备文件

- `SCENARIO.md`：场景定义与使用说明
- `templates.md`：可直接复制的模板集合
- `examples/`：脱敏输入/输出样例

### 可选文件

- `scripts/`：该场景专用脚本
- `assets/`：图片、表格头、附件模板、示意图等

### 命名要求

- 场景目录统一：`序号-英文短名`
- 序号两位：`01`、`02`、`03`
- 英文短名只描述核心任务，不要过长
- 不要使用空格
- 不要混用中文目录名和英文目录名

示例：

- `01-parent-daily-notice`
- `02-activity-notice`
- `11-exam-analysis`
- `25-monthly-class-report`

---

## 2. SCENARIO.md 排版规范

每个场景的 `SCENARIO.md` 必须按下面顺序组织：

```md
# 场景 01｜家长群日常通知

## 场景目标
一句话说明这个场景解决什么问题。

## 适用时机
- 什么时候适合用
- 哪些情况常用
- 哪些情况不建议用

## 输入
- 必填信息
- 可选信息
- 缺失时的默认处理方式

## 输出
- 默认输出什么
- 可选输出版本有哪些
- 输出是给老师看、给家长看，还是给系统用

## 推荐语气
- 正式
- 简洁
- 温和
- 催办
- 严肃
- 鼓励

## 文件说明
- `templates.md`：标准模板
- `examples/`：脱敏样例
- `scripts/`：如有自动化脚本，放这里

## 直接给 AI 的说法
- 示例 1
- 示例 2
- 示例 3

## 注意事项
- 默认先给草稿，不直接发送
- 不包含真实隐私
- 遇到敏感情形时需要人工确认
```

### 书写要求

- 一级标题只保留一个
- 二级标题固定，不随意换名字
- 每节内容尽量短，优先 bullet list
- 不写长篇概念解释
- 不写市场背景，不写路线图，不写与本场景无关内容

---

## 3. templates.md 排版规范

`templates.md` 的核心原则：**先给可复制模板，再给简短说明**。

推荐结构：

```md
# 模板｜家长群日常通知

## 模板 1｜标准正式版
**适用：** 常规通知  
**语气：** 正式、简洁  
**可替换字段：**
- [事项]
- [时间]
- [地点]
- [要求]

```text
各位家长好：

明天[事项]，请孩子于[时间]前到达[地点]。
请提前准备好[要求]，感谢配合。

班主任
[日期]
```

## 模板 2｜简洁提醒版
**适用：** 群内快速提醒  
**语气：** 简短、明确

```text
家长您好，提醒一下：明天[事项]，请孩子准备好[要求]，按时到[地点]。感谢配合。
```
```

### 模板文件强制要求

每个模板都必须有：

- 模板标题
- 适用说明
- 语气说明
- 可替换字段（如适用）
- 正文模板（必须放 code block）

### 禁止事项

- 不要把模板正文写成普通段落
- 不要把原理说明写在模板前面一大段
- 不要一个文件里混入大量散文式解释
- 不要出现真实姓名、电话、班级隐私、学校私密信息

---

## 4. examples/ 规范

`examples/` 用来告诉别人：

- 这个场景应该怎么喂数据
- 这个场景大概会输出什么结果

### 文件命名规范

推荐至少有一组：

```text
examples/
  input.sample.md
  output.sample.md
```

如果是结构化数据场景，可使用：

```text
examples/
  input.sample.csv
  input.sample.json
  output.sample.md
  output.sample.json
```

### 示例要求

- 必须脱敏
- 尽量短
- 一眼能看出输入与输出关系
- 不要放真实学生、家长、联系方式
- 不要把样例写成草稿堆积区

### 推荐做法

- 文案场景：输入用 `.md`，输出也用 `.md`
- 数据分析场景：输入用 `.csv` / `.json`，输出用 `.md` / `.json`
- 如果一个场景有多个输出风格，可命名为：
  - `output.formal.sample.md`
  - `output.short.sample.md`

---

## 5. scripts/ 规范

如果某个场景有脚本，脚本必须满足：

- 文件名表达用途
- README / SCENARIO.md 中有运行示例
- 输入格式明确
- 输出格式明确
- 不内嵌真实密钥、cookie、token、sessionKey

### 脚本文件建议结构

Python 脚本建议顶部包含：

```python
"""用途：用于班级成绩分析，输出 Markdown 报告。"""
```

同时在场景文档中写明：

- 输入文件要求
- 运行命令
- 输出文件示例
- 常见错误

---

## 6. assets/ 规范

`assets/` 只放辅助素材，例如：

- 表格模板截图
- PPT 页面示意
- 封面示意
- 可公开图片资源

不要放：

- 私有截图
- 带学生信息的原图
- 聊天记录截图
- 不可公开平台导出文件

---

## 7. 文件内容风格统一规则

### 说明类文件
如：`SCENARIO.md`、`README.md`

要求：

- 短段落
- 二级标题为主
- 优先列表，不堆长文
- 一段只说一件事

### 模板类文件
如：`templates.md`

要求：

- 一个模板一个小节
- 模板正文必须 code block
- 先模板，后说明
- 适用场景和语气必须写清楚

### 示例类文件
如：`examples/input.sample.*`

要求：

- 明确输入/输出身份
- 文件名一眼可懂
- 内容脱敏
- 示例尽量小而标准

### 脚本类文件
如：`scripts/*.py`

要求：

- 参数清楚
- 用法有例子
- 不写死敏感配置
- 输出可验证

---

## 8. 新增场景验收清单

新增一个场景前，至少满足：

- [ ] 有唯一场景编号
- [ ] 有清晰场景名称
- [ ] 有 `SCENARIO.md`
- [ ] 有至少 1 个可复制模板
- [ ] 有至少 1 组脱敏输入/输出示例
- [ ] 如有脚本，则写明运行方式
- [ ] 写清默认输出与边界
- [ ] 写清是否允许直接外发
- [ ] 不含真实隐私与敏感信息
- [ ] 文件命名与排版符合本规范

---

## 9. 对现有 class-assistant 的迁移建议

当前 `class-assistant` 已有：

- `references/`
- `examples/`
- `scripts/`

建议迁移节奏：

1. 先建立 `SCENARIOS_INDEX.md`，定义 25 个标准场景
2. 再逐步把 `references/` 中内容映射到具体场景
3. 以后新增内容优先按场景目录写，不再继续堆平铺式 references
4. `references/` 可保留为总资料库，但场景内容应逐步产品化
